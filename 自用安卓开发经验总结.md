# 一、Shizuku 核心经验

## 1.1 核心理念

> **Shizuku 的本质是给 APP 提权到 ADB shell 级别。既然有了 shell 权限，直接执行 shell 命令就是最正确的方式。**

不需要 Binder、不需要 AIDL、不需要反射调用 Shizuku 内部方法——直接通过 `newProcess("sh")` 获取 Shell 进程，写入命令即可。

## 1.2 ShizukuShell.java 最终实现（Java 反射）

```java
public class ShizukuShell {
    private static Method newProcessMethod;
    private static Class<?> shizukuClass;

    public static boolean isAvailable() {
        try {
            if (shizukuClass == null)
                shizukuClass = Class.forName("rikka.shizuku.Shizuku");
            Method ping = shizukuClass.getMethod("ping");
            return (boolean) ping.invoke(null);
        } catch (Exception e) { return false; }
    }

    public static boolean isGranted() {
        try {
            Class<?> c = Class.forName("rikka.shizuku.Shizuku");
            int r = (int) c.getMethod("checkSelfPermission").invoke(null);
            return r == 0;
        } catch (Exception e) { return false; }
    }

    public static void requestPermission(int requestCode) {
        try {
            Class<?> c = Class.forName("rikka.shizuku.Shizuku");
            c.getMethod("requestPermission", int.class).invoke(null, requestCode);
        } catch (Exception ignored) {}
    }

    public static ShellResult exec(String cmd) {
        try {
            Class<?> c = Class.forName("rikka.shizuku.Shizuku");
            Method m = c.getDeclaredMethod("newProcess",
                String[].class, String[].class, String.class);
            m.setAccessible(true);
            Process proc = (Process) m.invoke(null,
                new String[]{"sh"}, null, null);
            // 写入命令 + 退出码标记
            java.io.OutputStream os = proc.getOutputStream();
            os.write((cmd + " 2>&1\necho \"__EXIT__:$?\"\n").getBytes());
            os.write("exit\n".getBytes());
            os.flush();
            // 读取输出
            java.io.BufferedReader reader = new java.io.BufferedReader(
                new java.io.InputStreamReader(proc.getInputStream()));
            StringBuilder sb = new StringBuilder();
            String line;
            while ((line = reader.readLine()) != null) sb.append(line).append("\n");
            String output = sb.toString().trim();
            // 提取退出码
            int ec = proc.waitFor();
            java.util.regex.Matcher mat = java.util.regex.Pattern
                .compile("__EXIT__:(\\d+)").matcher(output);
            if (mat.find()) {
                try { ec = Integer.parseInt(mat.group(1)); } catch (Exception ignored) {}
            }
            String clean = output.replaceAll("__EXIT__:\\d+\\s*", "").trim();
            return new ShellResult(ec, clean);
        } catch (Exception e) {
            return new ShellResult(-1, "ERR: " + e.getMessage());
        }
    }

    public static class ShellResult {
        public final int exitCode;
        public final String output;
        public ShellResult(int ec, String out) { this.exitCode = ec; this.output = out; }
    }
}
```

### ⚠️ 反射参数传递陷阱

```java
// ✅ 正确写法（直接传参）
Process proc = (Process) m.invoke(null, new String[]{"sh"}, null, null);

// ❌ 错误写法（多包一层数组不会报错，但命令不执行）
Process proc = (Process) m.invoke(null, new Object[]{new String[]{"sh"}, null, null});
```

`Method.invoke(Object obj, Object... args)` 是**可变参数**，多包一层 `new Object[]{}` 是否真的会导致参数错乱取决于 Java 版本——在某些版本上 invoke 会解包数组，使得 `new Object[]{new String[]{"sh"}, null, null}` 实际上等价于分开传参。但为了保证一致性，**直接用分开传参的写法**。

### 1.3 反射类必须放 Java 文件

Kotlin 编译检查会拦截对 `private` 方法的反射调用，而 Java 不检查。所以 `ShizukuShell.java` 必须是 Java 文件。

### 1.4 管道写入模式

获取 Shell 后不要直接把参数传为数组，而是通过 shell 进程的 `getOutputStream()` 写入完整命令：

```java
Process proc = (Process) m.invoke(null, new String[]{"sh"}, null, null);
if (proc != null) {
    PrintWriter writer = new PrintWriter(proc.getOutputStream());
    writer.println(cmd);        // 写入完整命令
    writer.println("echo __EXIT__:$?");  // 标记退出码
    writer.flush();
    writer.close();
    // 读取 stdout ...
}
```

### 1.7 settings put secure 的 SELinux 差异

**重要发现**：同一个 `settings put secure` 命令在不同上下文表现不同：

| 执行环境 | vivo 设备上结果 |
|---------|--------------|
| `adb shell settings put secure` | ❌ 被 vivo 拦截，写入无效 |
| `ShizukuShell.exec("settings put secure ...")` | ✅ 成功 |

**根因**：vivo 设备的 SELinux 策略对 ADB shell 和 Shizuku 进程的约束不同。Shizuku (ADB mode) 的 `newProcess("sh")` 创建的进程有更宽松的安全上下文。

**实践**：不依赖 ADB 调试，直接在 App 内通过 ShizukuShell 执行：
```java
ShizukuShell.exec("settings put secure enabled_accessibility_services \"" + component + "\"");
ShizukuShell.exec("settings put secure accessibility_enabled 1");
```

# 二、无障碍服务（AccessibilityService）

## 2.1 核心机制

```
用户手动开启无障碍
      ↓
系统写入 settings secure:
  enabled_accessibility_services =  com.example/...Service
      ↓
手机重启 → SystemServer → AccessibilityManagerService 读取配置
      ↓
检查 android:enabled=true → 确认可用
检查 android:directBootAware=true → 无需等用户解锁
      ↓
系统自动创建 Service 实例 → onServiceConnected() 回调
```

**核心优势**：不写一行自启代码，全靠系统级保活。

## 2.2 AndroidManifest 声明（完整正确版）

```xml
<service
    android:name=".YourAccessibilityService"
    android:exported="true"
    android:enabled="true"
    android:directBootAware="true"
    android:permission="android.permission.BIND_ACCESSIBILITY_SERVICE">
    <intent-filter>
        <action android:name="android.accessibilityservice.AccessibilityService" />
    </intent-filter>
    <meta-data
        android:name="android.accessibilityservice"
        android:resource="@xml/accessibility_service_config" />
</service>
```

| 属性 | 作用 |
|------|------|
| `BIND_ACCESSIBILITY_SERVICE` | **必须**，确保只有系统能绑定该服务 |
| `exported="true"` | **必须**，system_server 跨进程绑定需要 |
| `enabled="true"` | 明确告知系统该组件已启用 |
| `directBootAware="true"` | 允许在解锁前（Direct Boot 模式）运行 |

### ⚠️ 踩坑：无障碍服务不显示在设置列表

**根因**：缺少 `BIND_ACCESSIBILITY_SERVICE` 权限 + `exported="false"`。

系统 `AccessibilityManagerService` 运行在 `system_server` 进程，必须跨进程绑定。如果服务没有声明 `BIND_ACCESSIBILITY_SERVICE`，系统拒绝绑定；如果 `exported="false"`，跨进程不可见。

**参考来源**：PermOpener 项目的 AndroidManifest 配置（已验证可工作）。

## 2.3 accessibility_service_config.xml

```xml
<accessibility-service xmlns:android="http://schemas.android.com/apk/res/android"
    android:accessibilityEventTypes="typeAllMask"
    android:accessibilityFeedbackType="feedbackGeneric"
    android:accessibilityFlags="flagReportViewIds|flagRetrieveInteractiveWindows"
    android:canRetrieveWindowContent="true"
    android:notificationTimeout="100" />
```

**关键 Flag**：

| Flag | 说明 |
|------|------|
| `canRetrieveWindowContent="true"` | **必须**，检索窗口内容总开关 |
| `flagReportViewIds` | 让节点携带 viewId 信息 |
| `flagRetrieveInteractiveWindows` | 访问所有交互窗口 |
| `typeAllMask` | 监听所有事件类型（方便开发，上线建议精简）|

## 2.4 Service 实现要点

```kotlin
class YourAccessibilityService : AccessibilityService() {

    override fun onServiceConnected() {
        super.onServiceConnected()
        // 动态设置 ServiceInfo
        val info = AccessibilityServiceInfo().apply {
            eventTypes = AccessibilityEvent.TYPE_VIEW_CLICKED or
                    AccessibilityEvent.TYPE_WINDOW_STATE_CHANGED
            feedbackType = AccessibilityServiceInfo.FEEDBACK_GENERIC
            flags = AccessibilityServiceInfo.FLAG_REPORT_VIEW_IDS
            notificationTimeout = 100
        }
        this.serviceInfo = info
    }

    override fun onAccessibilityEvent(event: AccessibilityEvent?) {
        if (event == null || !isRecording) return  // 用户手动控制开关
        when (event.eventType) {
            AccessibilityEvent.TYPE_VIEW_CLICKED -> {
                val (label, viewId) = extractClickInfo(event)
                // 记录点击
            }
            AccessibilityEvent.TYPE_WINDOW_STATE_CHANGED -> {
                val pkg = event.packageName?.toString()
                val screen = event.className?.toString()
                // 记录页面切换
            }
        }
        event.source?.recycle()  // 必须回收！
    }

    override fun onInterrupt() {}
}
```

### 5 级 Label 提取策略

```kotlin
private fun extractBestLabel(node: AccessibilityNodeInfo): String {
    // 1. contentDescription（TalkBack 主标签）
    node.contentDescription?.toString()?.takeIf { it.isNotBlank() }?.let { return it }
    // 2. text（按钮可见文字）
    node.text?.toString()?.takeIf { it.isNotBlank() }?.let { return it }
    // 3. hintText（输入框提示）
    node.hintText?.toString()?.takeIf { it.isNotBlank() }?.let { return it }
    // 4. viewId 资源名（最后一段）
    node.viewIdResourceName?.let { return it.substringAfterLast('/') }
    // 5. 子节点递归
    for (i in 0 until node.childCount) {
        node.getChild(i)?.let { child ->
            child.text?.toString()?.takeIf { it.isNotBlank() }?.let {
                child.recycle(); return it
            }
            child.recycle()
        }
    }
    return ""
}
```

## 2.5 安全编程原则

### ⚠️ 血泪教训：手机变砖的完整链路

```
手机开机
  ↓
系统自动启动无障碍服务（用户之前开启了）
  ↓
onAccessibilityEvent 被高频调用（开机过程系统UI变化极多）
  ↓
监听 TYPE_WINDOW_CONTENT_CHANGED → 递归遍历 UI 树
  ↓
叠加悬浮窗覆盖层 → 用户触摸屏幕无响应
  ↓
重启 → 再次进入死循环 → 只能 ADB 线连电脑解决
```

### 安全守则

```kotlin
// ❌ 不要在 onServiceConnected 中自动开启任何功能！
override fun onServiceConnected() {
    // enableOverlay() // ← 千万不要！
}

// ✅ 所有功能由用户手动触发
override fun onAccessibilityEvent(event: AccessibilityEvent?) {
    if (event == null || !recording) return  // ← recording 必须用户手动开启
}

// ❌ 不要监听高频事件做耗时操作 (TYPE_WINDOW_CONTENT_CHANGED)
// ✅ 只用低频事件 (TYPE_WINDOW_STATE_CHANGED, TYPE_VIEW_CLICKED)
// ❌ 不要用 START_STICKY（被杀后自动重启，危险！）
// ✅ 用 START_NOT_STICKY
```

### ADB 急救命令

```bash
# 清空无障碍服务列表（最有效）
adb shell settings put secure enabled_accessibility_services ""
# 停用无障碍功能
adb shell settings put secure accessibility_enabled 0
# 强制卸载
adb uninstall com.example.app
# 安全模式启动（第三方无障碍服务不启动）
adb reboot safe-mode
```

# 三、应用保活与开机自启

## 3.1 无障碍服务自启（核心方案）

**不需要 BOOT_COMPLETED 广播**。系统 `AccessibilityManagerService` 在开机时自动读取 `settings_secure.xml`，恢复已授权的无障碍服务。这是最可靠的自启方式。

## 3.2 前台服务保活

在 `onServiceConnected()` 中启动前台服务，常驻通知栏，显著降低被回收概率：

```kotlin
override fun onServiceConnected() {
    super.onServiceConnected()
    startForegroundService()
    // ... 业务逻辑
}

private fun startForegroundService() {
    val intent = Intent(this, KeepAliveService::class.java)
    if (Build.VERSION.SDK_INT >= Build.VERSION_CODES.O) {
        startForegroundService(intent)
    } else {
        startService(intent)
    }
}

class KeepAliveService : Service() {
    override fun onCreate() {
        super.onCreate()
        val notification = createNotification()
        startForeground(NOTIFICATION_ID, notification)
    }
    override fun onStartCommand(intent: Intent?, flags: Int, startId: Int): Int {
        return START_NOT_STICKY  // 安全
    }
}
```

class KeepAliveWorker(context: Context, params: WorkerParameters) : Worker(context, params) {
    override fun doWork(): Result {
        if (!isAccessibilityServiceRunning()) {
            showServiceDownNotification()  // 引导用户恢复
        }
        return Result.success()

    private fun isAccessibilityServiceRunning(): Boolean {
        val am = applicationContext.getSystemService(Context.ACCESSIBILITY_SERVICE) as AccessibilityManager
        val enabledServices = am.getEnabledAccessibilityServiceList(AccessibilityServiceInfo.FEEDBACK_ALL_MASK)
        val serviceId = "${applicationContext.packageName}/.YourAccessibilityService"
        return enabledServices.any { it.id == serviceId }

# 四、获取已安装应用列表

## 4.1 问题背景

Android 11（API 30）引入**包可见性限制**（Package Visibility）。第三方应用调用 `PackageManager.getInstalledApplications()` 时，如果不声明 `QUERY_ALL_PACKAGES`，系统只返回调用者自身的包信息。

## 4.2 不同方法对比

| 方法 | 结果 | 原因 |
|------|------|------|
| `queryIntentActivities(CATEGORY_LAUNCHER)` | ~20个 | 只返回桌面有图标的 |
| `pm list packages -3` (adb shell) | 104个 | shell UID，无限制 |
| `pm list packages -3` (run-as app) | 3个 | app UID，包可见性限制 |
| `getInstalledApplications()` (无权限) | 很少 | 受 QUERY_ALL_PACKAGES 限制 |
| `getInstalledApplications()` (QUERY_ALL_PACKAGES) | 全部 | 正常返回 |

## 4.3 最终方案

**targetSdk 32 + QUERY_ALL_PACKAGES 声明**，配合 `getInstalledApplications()` 获取全部应用，再用 `ApplicationInfo.FLAG_SYSTEM` 过滤系统应用：

```xml
<uses-permission android:name="android.permission.QUERY_ALL_PACKAGES"
    tools:ignore="QueryAllPackagesPermission" />
```

```kotlin
private fun loadInstalledApps(context: Context): List<AppInfo> {
    val pm = context.packageManager
    val allApps = mutableMapOf<String, String>()

    // 主方案：getInstalledApplications + QUERY_ALL_PACKAGES
    val flags = if (Build.VERSION.SDK_INT >= 33)
        PackageManager.ApplicationInfoFlags.of(0)
    else null
    val infos = if (flags != null)
        pm.getInstalledApplications(flags)
    else
        @Suppress("DEPRECATION") pm.getInstalledApplications(0)

    for (info in infos) {
        if (info.flags and ApplicationInfo.FLAG_SYSTEM != 0) continue  // 跳过系统应用
        val label = info.loadLabel(pm)?.toString() ?: info.packageName
        allApps[info.packageName] = label
    }

    return allApps.entries.map { AppInfo(it.key, it.value) }.sortedBy { it.label }
}
```

### 为什么不需要 QUERY_ALL_PACKAGES 也能查指定包名

- `getInstalledApplications()`：遍历全部包 → **需要** QUERY_ALL_PACKAGES
- `getApplicationInfo(指定包名, 0)`：精准查询已知包名 → **不需要** 任何额外权限

# 六、模拟定位开发

## 6.1 核心原理

### 三源同时推送

模拟定位不能只改 GPS，很多 App 会同时参考多个位置源：

```kotlin
lm.addTestProvider("gps", false, true, false, true, true, true, true, 0, 5)
lm.setTestProviderEnabled("gps", true)
lm.addTestProvider("network", ...)
lm.addTestProvider("passive", ...)
```

### 持续推送

```kotlin
while (isActive) {
    val location = Location("gps").apply {
        latitude = curLat + jitterLat
        longitude = curLng + jitterLng
        accuracy = 8f + jitterRand.nextFloat() * 12f  // 8~20m 真实精度
        speed = 0.5f + jitterRand.nextFloat() * 1.5f
        time = System.currentTimeMillis()
    }
    lm.setTestProviderLocation("gps", location)
    delay(100)
}
```

## 6.2 参数优化

| 参数 | 错误值 | 正确值 | 原因 |
|------|--------|--------|------|
| accuracy | 1米 | 8~20米随机 | 真实 GPS 精度在 5~20 米 |
| speed | 0（恒定） | 0.5~2.0 随机 | 静止也有微小波动 |
| 抖动 | 无 | ±10米随机 | 模拟 GPS 漂移 |
| 推送频率 | 仅1次 | 每100ms持续 | App 需要持续更新的位置 |

## 6.3 Provider 注册并发问题

`addTestProvider()` / `removeTestProvider()` **不是线程安全的**。必须用 `synchronized` 锁保护：

```kotlin
synchronized(mockLock) {
    stopMockInternal()
    if (!registerProviders(m)) {
        Handler(mainLooper).postDelayed({
            synchronized(mockLock) { registerProviders(m); startPushLoop() }
        }, 1000)
        return
    }
    startPushLoop()
}
```

## 6.4 自愈推送循环

```kotlin
private fun startPushLoop() {
    thread(start = true) {
        // 第一阶段：快速 burst 推送 3 次
        repeat(3) { pushOnce(); Thread.sleep(150) }
        // 验证系统是否采纳
        if (!isMockActive()) {
            ShizukuShell.exec("appops set $packageName android:mock_location allow")
            registerProviders(m)
            repeat(5) { pushOnce(); Thread.sleep(150) }
        }
        // 第二阶段：正常轮询（Handler 异步，不阻塞主线程）
        handler.post(object : Runnable {
            override fun run() {
                pushOnce()
                handler.postDelayed(this, 200)
            }
        })
    }
}
```

# 十一、万能APP疑难杂症

## 11.1 闪退：括号不匹配

**现象**：`apply_file` 导致花括号数量不等（`{52个}53个`），函数嵌套错位。

**解决**：`grep -o '{' file | wc -l` 和 `grep -o '}' file | wc -l` 定期检查匹配。

## 11.2 坐标从未推送到系统

**根因**：`registerProviders` 成功后没有 `return`，递归不停止。

## 11.3 主线程 sleep 阻塞

**根因**：`Thread.sleep` 阻塞 Handler 线程，导致 UI 更新延迟。

**修复**：改为 burstCount 连推模式（第一次推送 3 次，然后切换 Handler 异步轮询）。

## 11.4 第一次启动定位延迟

**根因**：Handler 第一轮 post 失败无重试机制。

**修复**：burst 推送 3 次 + 验证是否采纳 → 不采纳则重新授权 + 再推 5 次。

## 11.5 自动授权时序

**根因**：`AdbCommandActivity.onCreate` 中执行 Shizuku 命令，此时 Shizuku Binder 未就绪。

**修复**：移到 `MockLocationService.onStartCommand` 中执行。

## 11.6 编译 UP-TO-DATE

**现象**：在 `/tmp/` 修改代码后，Gradle 检测不到源码变化。

**解决**：先 `rm -rf /tmp/project` 再重新 `cp` 编译。

# 十二、通用经验教训

## 12.1 核心原则

> **代码结构比逻辑更重要**：括号不匹配、函数嵌套错了，逻辑再对也跑不起来

1. **先查资料再动手** — 出问题先看 logcat 确认异常类型 → 搜解决方案 → 确认 root cause 后动手
2. **不要闭门造车** — 你在会遇到的问题，别人一定已经遇到并解决了
3. **每一步都验证** — 改一行代码编译一次，不等到最后一次性验证

## 12.2 代码检查清单

- [ ] 反射调用参数传递方式是否正确（Method.invoke 可变参数）
- [ ] 异常不要完全吞掉，最少输出到 logcat
- [ ] 递归函数是否有明确的终止条件
- [ ] 花括号数量是否匹配：`grep -o '{' file | wc -l` = `grep -o '}' file | wc -l`
- [ ] 函数嵌套是否正确：`grep -n "private fun\|fun "` 检查函数定义位置
- [ ] `apply_file` / `Edit` 大段替换是否漏掉或破坏了相邻代码

## 12.3 调试命令

```bash
# 崩溃日志
adb logcat -b crash -d

# 指定 Tag
adb logcat -s "ActionPilot" -d

# 模拟定位状态
dumpsys location | grep -A5 "gps provider \[mock\]"

# 无障碍服务配置
settings list secure | grep accessibility

# 当前焦点窗口
dumpsys window | grep mCurrentFocus

# 包信息
dumpsys package <pkg>
```

## 12.4 Shizuku 专区

1. 反射调用 newProcess 必须放 Java 文件（Kotlin 编译检查拦截 private 方法）
2. `AppOpsService.setMode()` 权限不够，用 shell 命令替代
3. 不要搞 Binder transact 或 UserService，shell 命令就是最直接的方式
4. Shizuku 绑定时序：**Service.onStartCommand 比 Activity.onCreate 更可靠**
5. 异常不要完全吞掉，最少输出到 logcat

## 12.5 文件与存储

1. 文件写入必须用 `filesDir`（内部存储），写 `/sdcard/` 在 Android 10+ 可能被 SELinux 拦截
2. 先写缓存文件再 rename，防止写坏
3. APK 从 `/sdcard/` 安装被 SELinux 限制 → 从 `/data/local/tmp/` 安装

## 12.6 无障碍服务安全

1. **全部功能用户手动触发，绝不在 onServiceConnected 中自动启动**
2. **只监听低频事件**：TYPE_WINDOW_STATE_CHANGED / TYPE_VIEW_CLICKED
3. **绝不监听 TYPE_WINDOW_CONTENT_CHANGED**（频率极高，可导致死循环变砖）
4. **用 START_NOT_STICKY**，防止被杀后自动重启
5. 开发时连好 ADB，写好卸载脚本，备用机测试
